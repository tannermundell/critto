# Phase 1 — Architecture & Requirements

**Project:** Critto
**Scope:** Image recognition of South African birds, mammals, and reptiles, with a grounded AI field-guide agent. Built for the AMD Developer Hackathon: ACT II.
**Status:** Draft v1 — for review and iteration.

Related: see `Animal-ID-Build-Scope.md` for the product framing and phasing.

---

## 1. Purpose & goals

Build a working, demo-ready app where a user uploads a photo of a SA animal, gets an accurate species ID, and receives a designed, factually-grounded field-guide entry generated by an AI agent — with the vision model on the AMD GPU (ROCm) and the agent LLM on Fireworks AI (AMD-hosted). Submitted to **Track 3 (Unicorn)** as a product/startup project.

**Success criteria**

- A judge can upload a photo and get a correct ID + rich entry in one smooth flow.
- Vision inference runs on the AMD GPU (ROCm); the agent LLM runs on Fireworks AI.
- The app is containerized, in a public GitHub repo with a README, and runnable from those instructions.
- The result card looks polished enough to anchor the 2-minute demo video.
- Strong **top-3 accuracy** on the curated species set (top-1 is a stretch goal — real photos are messy).

**How this maps to Track 3 judging**

- *Creativity & originality* — localised SA focus + a grounded field-guide agent, not a generic classifier.
- *Product/market potential* — SA tourism, safari, birding, conservation/education.
- *Completeness* — a polished, working end-to-end demo.
- *Use of AMD platforms* — AMD on two fronts: vision on the AMD GPU (ROCm) + agent LLM on Fireworks AI.

## 2. Locked decisions

| Decision | Choice | Why |
|---|---|---|
| Species set size | **~100–150 species** across the three classes | Meaningful breadth without blowing the time/accuracy budget |
| LLM hosting | **Fireworks AI API** (AMD-hosted models) | Satisfies "use of AMD platforms" with low effort; frees the scarce GPU window for the vision model only |
| Auth & persistence | **Light Supabase auth + persistent sightings log** | Per-user "life list" makes a richer demo and adds stickiness |

## 3. Functional requirements

**Capture & identify**

- FR1 — User can upload an image or capture one via device camera.
- FR2 — System returns the **top-N candidate species** (target N=3) with confidence scores.
- FR3 — If top-1 confidence is high, surface it directly; otherwise present the candidates for the user to choose.
- FR4 — User confirms or selects the correct species.

**Field-guide entry**

- FR5 — On confirmation, the knowledge agent returns a structured entry: common + scientific name, IUCN conservation status, habitat, diet, size, range, "where to spot it," and a fun fact.
- FR6 — Entry facts are **retrieved from real sources** (not freely generated) and the agent cites them.
- FR7 — Entries are cached per species so repeat lookups are instant and cheap.

**Account & history**

- FR8 — User can sign up / log in (Supabase auth).
- FR9 — Confirmed sightings are saved to the user's personal log with the photo, species, date, and (optional) location.
- FR10 — User can browse their sightings log ("life list").

**Result card**

- FR11 — A designed result screen presents the entry with the species image, status badge, and key facts.
- FR12 — Proper empty, loading, and error states.

---

## 4. Non-functional requirements

- NFR1 — **Latency:** `/identify` target < ~3s; `/entry` < ~10s on first generation, instant when cached.
- NFR2 — **Accuracy:** prioritise top-3 accuracy; track per-class (expect reptiles weakest).
- NFR3 — **Cost / credits:** single GPU instance hosting both models; instance up only for dev sessions and the demo. Budget against the $100 credit pool.
- NFR4 — **Demo reliability:** maintain a set of known-good test images and a fallback path so a live demo can't hard-fail.
- NFR5 — **Decoupling:** the inference server, backend, and frontend communicate over clean APIs so Phase 2 (sound) and Phase 3 (beyond SA) bolt on without a rewrite.
- NFR6 — **Security:** inference endpoint protected by an auth token; uploaded images access-controlled in Supabase Storage.
- NFR7 — **Containerization:** the inference server ships as a Docker container; public GitHub repo with a README; the app runs from those instructions (submission requirement).

## 5. System architecture

Three deployable pieces around one principle: **the AMD GPU does the core compute.**

**Frontend — Lovable**
- Mobile-first web app: capture/upload, candidate selection, result card, sightings log.
- Talks to Supabase directly for auth, storage, and data.

**Backend / data — Supabase**
- Auth, Postgres (species + sightings), Storage (uploaded images), edge functions to broker calls to the GPU server.

**Inference + agent server — containerized Python (FastAPI)**
- `/identify`: the vision model, running on the AMD Developer Cloud GPU (PyTorch on ROCm).
- `/entry`: the knowledge agent, which calls the LLM on Fireworks AI (AMD-hosted).
- Ships as a Docker container.

```
┌──────────────┐      ┌──────────────────┐      ┌────────────────────────────┐
│   Lovable    │      │    Supabase      │      │  AMD Developer Cloud GPU   │
│  (frontend)  │      │ (backend / data) │      │ (FastAPI + PyTorch/ROCm)   │
│              │      │                  │      │                            │
│ upload/snap  │──1──▶│ storage + DB     │──2──▶│ /identify (vision model)   │
│ result card  │◀─────│ edge functions   │◀─3───│   → top-N + confidence     │
│ confirm pick │──4──▶│                  │      │                            │
│ sightings log│      │                  │──5──▶│ /entry (knowledge agent)   │
│              │◀─────│ save sighting    │◀─6───│   → grounded field guide   │
└──────────────┘      └──────────────────┘      └────────────────────────────┘
```

## 6. Component specs

### 6.1 Vision model (`/identify`)

- **Approach:** start with an open biological vision model (e.g. BioCLIP) evaluated zero/few-shot against the ~100–150 SA labels. If top-3 accuracy is weak, **fine-tune** a vision model (ViT/ResNet class) on SA observations — very feasible at this class count.
- **Input:** image (signed URL or base64). **Output:** ranked candidates `[{label, scientific_name, confidence}]`.
- **Verify early:** ROCm compatibility and that the chosen model loads/runs on the AMD instance.

### 6.2 Knowledge agent (`/entry`)

- **LLM on Fireworks AI** (AMD-hosted; specific model chosen from those revealed on launch day). The server holds the Fireworks API key and makes the calls.
- **Agentic loop:** species in → call retrieval tools (Wikipedia, IUCN Red List) → synthesise structured, grounded fields → return with sources.
- **Guardrail:** the LLM formats retrieved facts; it does not invent conservation status, range, or other hard facts.
- **Caching:** store generated entries against the species record; regenerate only on demand.

### 6.3 Frontend (Lovable)

- Screens: capture/upload, "we think it's…" candidate selection, result card, sightings log, auth.
- Result card is the hero — design priority.

### 6.4 Backend (Supabase)

- Auth, DB, Storage, and edge functions that call the GPU server (holding the GPU auth token server-side, never in the client).

## 7. Data model (Supabase / Postgres)

**`species`** (curated, ~100–150 rows)

| field | type | notes |
|---|---|---|
| id | uuid | PK |
| common_name | text | |
| scientific_name | text | |
| class | enum | bird / mammal / reptile |
| model_label | text | maps to classifier output |
| iucn_status | text | from IUCN |
| habitat | text | |
| range_desc | text | |
| reference_image_url | text | licensing checked |
| cached_entry | jsonb | agent output, nullable |

**`sightings`**

| field | type | notes |
|---|---|---|
| id | uuid | PK |
| user_id | uuid | FK → auth.users |
| species_id | uuid | FK → species, nullable until confirmed |
| image_url | text | Supabase Storage |
| confidence | float | top candidate score |
| candidates | jsonb | full top-N returned |
| location | geography | optional, nullable |
| created_at | timestamptz | |

Users handled by Supabase auth.

## 8. API contract (GPU server)

**`POST /identify`**
- Request: `{ "image_url": "<signed url>" }`
- Response: `{ "candidates": [ { "label": "...", "scientific_name": "...", "confidence": 0.0 } ], "model_version": "..." }`

**`POST /entry`**
- Request: `{ "species": { "common_name": "...", "scientific_name": "...", "class": "bird" } }`
- Response: `{ "status": "...", "habitat": "...", "diet": "...", "size": "...", "range": "...", "where_to_spot": "...", "fun_fact": "...", "sources": [ "..." ] }`

Both endpoints require the shared auth token.

## 9. Data sources

- **Species list + reference images:** iNaturalist / GBIF (SA observations).
- **Conservation status:** IUCN Red List.
- **Field-guide facts (agent retrieval):** Wikipedia, IUCN.
- **Action:** script the species-list assembly (pull SA species per class, join IUCN status) rather than hand-curating — this is the biggest hidden time sink.

## 10. Risks & mitigations

| Risk | Mitigation |
|---|---|
| ROCm incompatibility (vision model or LLM serving) | Test both on the AMD instance on Day 1, before building anything else |
| Both models exceed GPU VRAM | Confirm instance VRAM; size the LLM to fit alongside the vision model; quantise if needed |
| Reptile accuracy poor | Use a smaller, well-separated reptile subset; lean on top-3; set expectations |
| Species-list curation eats time | Script the pull from GBIF/iNat + IUCN instead of manual curation |
| Image licensing for cards | Verify reuse rights before redistributing reference images |
| GPU credits burn while idle | Bring the instance up only for dev sessions and the demo |
